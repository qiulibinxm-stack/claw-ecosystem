# Claude Code记忆系统解密

> 来源：openclaw-control-ui | 日期：2026-04-21

## 一句话摘要

Claude Code的记忆系统采用"4+1"架构（四层配置+AutoDream），核心是**双轨反馈记录**+**三重压缩**+**生物模拟整理**，形成可控、可审、低消耗的记忆闭环。

---

## 核心架构：4+1结构

```
层级         位置              共享范围        优先级
─────────────────────────────────────────────────────
1. 公司配置   云端             全公司共享       低（基础）
2. 个人全局   ~/.claude/       跨项目生效       中
3. 仓库共享   .claude/         团队共享         高
4. 本地私有   .claude.local/   不提交           最高

加载顺序：1→4（逐层覆盖）
执行优先级：4→1（后加载覆盖先加载）
```

## 关键机制

### 1. 自动记忆Agent
- **后台独立运行**，分类存储：
  - 用户身份信息
  - 纠正记录
  - 项目上下文
  - 外部资源
- **双轨记录**：纠正+认可反馈都存
- **时间标准化**：相对时间→绝对时间戳（解决时间漂移）

### 2. 三重压缩机制
| 压缩级别 | 触发条件 | 保留内容 |
|----------|----------|----------|
| 微压缩 | 日常运行 | 完整对话 |
| 自动压缩 | 剩余Token < 13,000 | 摘要+关键对话 |
| 完整压缩 | 严重超限 | 仅保留修改记录+反馈+最近对话 |

**熔断机制**：连续3次失败→停止重试

### 3. AutoDream模块（模拟人脑整理）
**触发条件**：
- 距上次整理 ≥ 24小时
- 期间 ≥ 5个工作session

**四阶段流程**：
```
定向 → 收集 → 整合 → 剪枝
```
**关键操作**：
- 相对时间→绝对时间（防止记忆漂移）
- 维持单文件≤200行
- 解决重复/冲突记忆

## 设计哲学

| 原则 | 具体实现 |
|------|----------|
| 控制感优先 | 纯文件系统，人工可读可改，Git版本管理 |
| 成本意识 | 极致优化Token消耗（13,000阈值触发压缩） |
| 生物模拟 | 模仿人脑记忆整理（睡觉=做梦=整理） |
| 产品壁垒 | 记忆系统构成AI核心差异化能力 |

## 对比：Claude Code vs OpenClaw

| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| 记忆结构 | 4+1（四层配置+AutoDream） | MEMORY.md + memory/日期.md |
| 压缩机制 | 三重压缩+熔断 | LCM（Lossless Context Management） |
| 整理机制 | AutoDream（24h/5session触发） | dream skill（手动触发） |
| 文件限制 | 单文件≤200行，每行≤150字符 | 无硬性限制 |
| Git管理 | 全部可版本追踪 | 部分可追踪 |
| 双轨反馈 | 纠正+认可都记录 | 主要记录纠正和关键事件 |

## 可借鉴点

1. **时间标准化**：相对时间→绝对时间戳（我们memory/日期.md中已部分实现）
2. **双轨反馈**：不仅记纠正，也记认可（用户说"对""好的"→记录为正面确认）
3. **熔断机制**：连续失败停止重试，防止无限循环
4. **AutoDream触发条件**：24h/5session可量化，可引入HEARTBEAT检查
5. **文件限制**：单文件≤200行，强制剪枝，防止记忆膨胀
