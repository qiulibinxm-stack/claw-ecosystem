# 八字排盘：系统错误诊断与修复全记录

> 记录时间：2026-04-13

## 问题现象

用户两次反馈：
1. **万年历换算出错** — 农历1985年2月21日戌时，多次换算结果错误，偏差近一个月
2. **四柱一直弄错** — 排盘结果和实际八字不符

## 根因诊断过程

### 第一步：跑通全链路数据

写诊断脚本 `diag2.py`，系统验证：
- 农历→公历：`lunardate` 库给出 1985年2月21日 → 1985-04-10 ✅ 正确
- 日柱手算 vs cnlunar：`lunardate`锚点推算己卯 = cnlunar返回己卯 ✅ MATCH
- 月柱节气边界：清明4月5日后进入辰月 ✅ cnlunar正确

**结论：lunardate 和 cnlunar 库本身计算正确，问题在 bazi skill 的使用方式。**

### 第二步：对比日志记录找差异

| 项目 | 日志记录 | cnlunar计算 | 正确性 |
|------|---------|------------|--------|
| 农历→公历 | 4月10日 | 4月10日 | ✅ 一致 |
| 月柱 | 辛卯 | 庚辰 | ❌ 日志错 |
| 日柱 | 戊辰 | 己卯 | ❌ 日志错 |

**日志记录的辛卯 = 农历二月 = 卯月（错在用农历月份算月柱）**

### 第三步：定位根因

1. **bazi skill 的 SKILL.md** 告诉 AI 可以"参照规则自行推算"，导致 AI 凭记忆推算
2. AI 错误地用农历月份（二月=卯月）确定月柱，而正确的做法是：
   - 先将农历转为公历（1985-04-10）
   - 判断节气：清明（4月5日）已过 → 辰月（三月）
   - 用年上起月法：乙年"乙庚之岁戊为头" → 三月=**庚辰**
3. bazi skill 没有强制调用 bazi_calculator.py 脚本

## 核心易错点

### 1. 月柱用节气，不用农历月份

| 农历日期 | 错误做法 | 正确做法 |
|---------|---------|---------|
| 农历1985年2月21日 | 二月=卯月→辛卯 ❌ | 清明4月5日后=辰月→庚辰 ✅ |

口诀：**节气定月柱，农历月份是干扰项**

### 2. 年柱用立春，不用农历正月初一

| 日期 | 年柱 |
|------|------|
| 1985-02-03（农历腊月十五，立春前） | 甲子（属1984年） |
| 1985-02-04（农历腊月十六，立春后） | 乙丑（属1985年） |

### 3. 日柱必须查库，不手算

六十甲子循环，日柱每天一变。必须用 cnlunar 库精确获取，严禁心算或用"经验公式"。

## 修复方案

### 1. 修复 bazi_calculator.py

- 修正 `LunarDate` 闰月参数名（`leap` → `isLeapMonth`）
- 修正 `cnlunar` API 调用（`isLeapMonth` → `isLunarLeapMonth`）
- 支持多种时辰输入格式（戌时 / 20:30 / HH）
- 支持闰月农历输入
- 内置结果校验
- 友好的格式化输出

### 2. 重写 bazi skill SKILL.md

- **强制要求**：所有排盘必须调用 bazi_calculator.py
- **高亮警示**：月柱用节气不用农历月份
- 移除所有让 AI 自行推算四柱的指引

### 3. 重写 bazi-master SKILL.md

- 明确工具定位（只做排盘，不做分析）
- 提供完整的 Python 和命令行调用示例
- 内置验证案例表

## 教训总结

1. **八字排盘必须用经过验证的库**：lunardate + cnlunar，手算几乎必错
2. **节气和农历月份是两套完全不同的系统**：八字只用节气
3. **bazi skill 的 SKILL.md 设计缺陷**：没有强制要求调用脚本，导致 AI 自由发挥
4. **PowerShell 编码问题**：Windows PowerShell 6.0 以下默认 GBK 编码，中文输出会乱码（文件写入不受影响）
